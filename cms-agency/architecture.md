# CMS Agency Starter Kit — Architecture Document

> **Version:** 1.0  
> **Goal:** Production-ready, single-tenant CMS for custom service-business websites. No databases. All content lives in JSON files on disk. The developer controls layout and sections; the client only edits content via a polished React admin dashboard.

## 2. JSON Schema Design

### 2.1 Pages (`content/pages/<slug>.json`)

Each page is a JSON document with a `sections` array. The CMS reads/writes this directly; the frontend website also reads it to render the live site.

```json
{
  "slug": "home",
  "title": "Home",
  "meta": {
    "title": "Welcome to Our Practice | Dr. Smith Dentistry",
    "description": "Best dental care in Springfield...",
    "canonical": "https://example.com/",
    "og": {
      "title": "...",
      "description": "...",
      "image": "/uploads/hero.jpg"
    },
    "schema": "{ \"@context\": \"https://schema.org\", ... }"
  },
  "sections": [
    {
      "id": "sec-abc123",
      "type": "hero",
      "fields": {
        "heading": "Welcome to Our Practice",
        "subheading": "Family dentistry you can trust.",
        "buttonText": "Book Now",
        "buttonLink": "/contact",
        "image": "/uploads/hero.jpg"
      }
    },
    {
      "id": "sec-def456",
      "type": "services",
      "fields": {
        "heading": "Our Services",
        "showIcons": true,
        "selectedServices": ["srv-1", "srv-3"]
      }
    }
  ]
}
```

- `sections` is ordered. Reordering writes back in new order.
- `id` is a UUID or nanoid generated by the CMS when a section is added.
- `type` is the section key that the developer maps to a layout component.
- `fields` are arbitrary key-value pairs defined by the developer's section schema. The CMS never assumes what fields exist; it reads the schema from the dev config.

### 2.2 Collections (`content/collections/<name>.json`)

Collections are arrays of items with a flat, consistent schema.

```json
{
  "name": "testimonials",
  "items": [
    {
      "id": "tst-1",
      "createdAt": "2024-01-15T10:00:00Z",
      "updatedAt": "2024-02-01T14:00:00Z",
      "fields": {
        "name": "Jane D.",
        "role": "Patient",
        "quote": "The staff was amazing!",
        "image": "/uploads/jane.jpg",
        "rating": 5
      }
    }
  ]
}
```

Supported collections: `testimonials`, `faqs`, `services`, `team`, `blog-posts`.

**Blog posts** use the same structure but include a `status` field:

```json
{
  "id": "bp-1",
  "status": "published",
  "publishedAt": "2024-03-10T09:00:00Z",
  "fields": {
    "title": "5 Tips for Dental Health",
    "slug": "5-tips-for-dental-health",
    "excerpt": "...",
    "content": "...markdown or plain text...",
    "featuredImage": "/uploads/blog1.jpg",
    "category": "Health",
    "tags": ["teeth", "prevention"]
  }
}
```

### 2.3 Settings (`content/settings/<name>.json`)

Flat key-value documents for global configuration.

```json
// business.json
{
  "name": "Dr. Smith Dentistry",
  "phone": "(555) 123-4567",
  "email": "hello@drsmith.com",
  "address": "123 Main St, Springfield",
  "hours": "Mon-Fri 9am-5pm"
}

// seo.json
{
  "defaultTitle": "Dr. Smith Dentistry",
  "defaultDescription": "...",
  "googleAnalyticsId": "G-XXXX",
  "favicon": "/uploads/favicon.png"
}

// branding.json
{
  "logo": "/uploads/logo.svg",
  "primaryColor": "#2563eb",
  "secondaryColor": "#1e40af"
}
```

## 3. API Architecture (Express)

### Base URL: `/api`

| Method | Endpoint | Description | Auth |
|--------|----------|-------------|------|
| POST | `/api/auth/login` | Login, set session cookie | Public |
| POST | `/api/auth/logout` | Destroy session | Required |
| GET | `/api/auth/me` | Check current session | Required |
| GET | `/api/pages` | List all page files | Required |
| GET | `/api/pages/:slug` | Read a page JSON | Required |
| PUT | `/api/pages/:slug` | Write a page JSON | Required |
| GET | `/api/collections/:name` | Read a collection JSON | Required |
| PUT | `/api/collections/:name` | Write a collection JSON | Required |
| GET | `/api/settings/:name` | Read a settings JSON | Required |
| PUT | `/api/settings/:name` | Write a settings JSON | Required |
| GET | `/api/media` | List all files in uploads/ | Required |
| POST | `/api/media/upload` | Upload image(s) | Required |
| DELETE | `/api/media/:filename` | Delete an image | Required |
| GET | `/api/preview/:slug` | Alias for `GET /api/pages/:slug` | Required |

**Notes:**
- All file I/O is synchronous (`fs.readFileSync` / `fs.writeFileSync`) for simplicity and immediate consistency. We use `fs-extra` for safety.
- All JSON writes are atomic: write to a temp file first, then rename to the target.
- The API uses a custom `authMiddleware` that checks `req.session.userId`. If missing, return `401`.
- CORS is enabled for local dev, but session cookies are `httpOnly` and `sameSite: 'strict'`.

### Error Handling

A centralized `errorHandler` middleware catches all errors and returns a consistent JSON shape:

```json
{ "success": false, "message": "Page not found", "code": "PAGE_NOT_FOUND" }
```

### Tech Stack
- **React 18** with functional components + hooks
- **Vite** for build tooling

| `/admin/seo` | `SeoManagerPage` | SEO |

| `/admin/login` | `LoginPage` | — |

### Layout

```
┌─────────────────────────────────────────┐
│  Header: Logo + User Menu + Logout      │
├──────────┬──────────────────────────────┤
│          │                              │
│ Sidebar  │  Main Content Area           │

└──────────┴──────────────────────────────┘
```

- Sidebar is collapsible on mobile (hamburger menu).
- All pages are protected by `AuthGuard` wrapper that checks `/api/auth/me` and redirects to `/admin/login` if not authenticated.

- **AuthContext**: Provides `user`, `login(credentials)`, `logout()`, and `isLoading`.
- **useApi Hook**: Wraps Axios. Handles auth headers implicitly (session cookie), loading state, and errors. Returns `{ data, loading, error, execute }`.

## 5. Section System Architecture

This is the core differentiator. The developer defines what sections are available and what fields each section has. The CMS reads this definition to generate the UI.

### 5.1 Developer Config (`client/src/config/sections.js`)

```js
export const sectionRegistry = {
  hero: {
    label: "Hero Section",
    icon: "Star",
    fields: [
      { name: "heading", type: "text", label: "Heading", required: true },
      { name: "subheading", type: "textarea", label: "Subheading" },
      { name: "buttonText", type: "text", label: "Button Text" },
      { name: "buttonLink", type: "text", label: "Button Link" },
      { name: "image", type: "image", label: "Background Image" }
    ]
  },
  services: {
    label: "Services List",
    icon: "Briefcase",
    fields: [
      { name: "heading", type: "text", label: "Section Heading" },
      { name: "showIcons", type: "boolean", label: "Show Icons?" },
      { name: "selectedServices", type: "relation", label: "Select Services", collection: "services" }
    ]
  },
  testimonials: {
    label: "Testimonials",
    icon: "MessageCircle",
    fields: [
      { name: "heading", type: "text", label: "Heading" },
      { name: "testimonialIds", type: "relation", label: "Select Testimonials", collection: "testimonials", multiple: true }
    ]
  },
  // ... faq, gallery, team, cta, contact, about, etc.
};
```

### 5.2 Field Types

The CMS supports a fixed set of field types. Adding a new type requires updating the `SectionForm` component, but the system covers 95% of use cases out of the box.

| Type | UI Component | Value Stored |
|------|--------------|--------------|
| `text` | `<Input>` | String |
| `textarea` | `<Textarea>` | String |
| `richText` | Simple WYSIWYG (contenteditable wrapper) | HTML String |
| `number` | `<Input type="number">` | Number |
| `boolean` | Toggle switch | Boolean |
| `image` | `<ImagePicker>` | String (URL path) |
| `select` | `<Select>` | String |
| `multiSelect` | Checkboxes or Tag selector | Array of strings |
| `relation` | `<RelationPicker>` (select items from a collection) | String ID or Array of IDs |
| `color` | `<input type="color">` | Hex string |
| `url` | `<Input type="url">` | String |
| `date` | `<Input type="date">` | ISO Date string |

### 5.3 Section Builder UI Flow

The `PageEditorPage` renders a list of sections for the current page.

```
[ + Add Section ]  ← opens a modal listing all available section types

1. Hero Section
   ┌──────────────────────┐
   │ Heading: [__________] │
   │ Subheading: [_______] │
   │ Image: [Select Image] │
   │ [Duplicate] [Delete]  │
   └──────────────────────┘
   [↑] [↓]  ← reorder

2. Services Section
   ...
```

- **Add**: User picks a section type from the modal. The CMS appends a new section with default field values (empty strings or empty arrays) and a new `id`.
- **Delete**: Confirmation modal, then removes from the array.
- **Duplicate**: Deep-clones the section object, generates a new `id`, and inserts it immediately after the original.
- **Reorder**: Up/down buttons move the section in the array. (Future: drag-and-drop via `@hello-pangea/dnd` if needed; for now, buttons keep it dependency-free.)
- **Save**: The entire page JSON is sent to `PUT /api/pages/:slug`.

### 5.4 Frontend Website Integration

The developer's website reads `content/pages/home.json` and maps `section.type` to a React component.

```jsx
// In the client's website frontend
import { HeroSection } from "./sections/HeroSection";
import { ServicesSection } from "./sections/ServicesSection";

const sectionMap = {
  hero: HeroSection,
  services: ServicesSection,
  // ...
};

export function Page({ pageData }) {
  return (
    <main>
      {pageData.sections.map((sec) => {
        const Component = sectionMap[sec.type];
        return Component ? <Component key={sec.id} fields={sec.fields} /> : null;
      })}
    </main>
  );
}
```

The CMS does not render the client's website. It only provides the JSON data. The developer builds the website separately using Next.js, Vite, or even plain HTML. The CMS is a data layer.

## 6. Authentication Architecture

### 6.1 Backend

- **express-session** with `MemoryStore` (sufficient for a single-tenant CMS with 1-5 users). No Redis needed.
- Session config:
  - `secret`: loaded from `process.env.SESSION_SECRET` or a generated fallback (warned in logs).
  - `cookie`: `httpOnly: true`, `secure: process.env.NODE_ENV === 'production'`, `sameSite: 'strict'`, `maxAge: 24h`.
- **Password Storage**: A single admin user is stored in `content/settings/auth.json`:
  ```json
  { "username": "admin", "passwordHash": "bcrypt_hash..." }
  ```
  On first run, if the file is missing, the server auto-generates a user with `admin` / `changeme` and logs a warning to the console.
- **Login Flow**: `POST /api/auth/login` → `bcrypt.compare` → `req.session.userId = 'admin'` → `200 OK`.
- **Logout Flow**: `POST /api/auth/logout` → `req.session.destroy()` → `200 OK`.
- **Auth Middleware**: `req.session.userId` must exist. Returns `401` otherwise.

### 6.2 Frontend

- `AuthContext` wraps the app.
- On app mount, it calls `GET /api/auth/me`. If `200`, user is authenticated. If `401`, redirect to `/admin/login`.
- `LoginPage` captures credentials, calls `POST /api/auth/login`, updates `AuthContext`, and redirects.
- Logout button clears the context and calls `POST /api/auth/logout`.

## 7. Media Library Architecture

### 7.1 Backend

- `multer` handles file uploads in `server/src/routes/media.js`.
- Files are saved to `content/uploads/`. The filename is sanitized and appended with a timestamp to avoid collisions: `hero-1709123456789.jpg`.
- `GET /api/media` reads the `content/uploads/` directory and returns metadata: `{ filename, url, size, createdAt }`.
- `DELETE /api/media/:filename` removes the file and also scans all JSON files in `content/` to remove references. (This is a "safe delete" — if the file is referenced anywhere, we either block deletion or warn the user. For MVP, we will block deletion if referenced and return a 400 error with the list of references.)

### 7.2 Frontend

- `MediaLibraryPage` displays a grid of thumbnails.
- Features: Upload (drag-and-drop), Delete, Search (client-side filter by filename), Copy URL (clipboard API).
- `ImagePicker` component (used inside `SectionForm`) opens a modal overlay of the Media Library. When the user clicks an image, the modal closes and the field value is updated to the image URL.

## 8. Live Preview System

The goal is to let the client see unsaved changes before they hit "Save."

### 8.1 Approach: In-App Preview Pane

The `PageEditorPage` has a "Preview" tab/button.
- When the user clicks Preview, the app opens a **side-by-side preview** or a **new browser tab**.
- The preview URL is the *client's actual website*, but with a query parameter `?cms_preview=1&data=<base64_encoded_page_json>`.
- The developer's website frontend checks for `?cms_preview=1`. If present, it reads the `data` param, decodes it, and uses that JSON instead of fetching the static JSON file.

**Why this approach?**
- It does not require the CMS to render the website. The developer's website already knows how to render the sections.
- It works instantly with no server-side complexity.
- It is safe because the preview is purely client-side.

### 8.2 Implementation Details

- CMS Side: The `PageEditorPage` has a "Preview" button that opens `window.open` to the client's website URL (read from `settings/business.json` or a `config.js` in the CMS) with the current, unsaved page state encoded as a base64 query string.
- Website Side: The developer adds a tiny snippet to their website's page loader:
  ```js
  if (window.location.search.includes('cms_preview=1')) {
    const params = new URLSearchParams(window.location.search);
    const json = JSON.parse(atob(params.get('data')));
    renderPage(json);
  } else {
    fetch('/content/pages/home.json').then(renderPage);
  }
  ```

- Note: Base64 encoding a JSON file might hit URL length limits for very large pages (e.g., 50KB+). For the MVP, we assume pages are small enough. If a page is too large, we can switch to `POST` via a temporary preview endpoint, but that's not required for the initial architecture.

## 9. Developer Experience (DX)

### 9.1 Configuration-Driven UI

The CMS is fully driven by developer config files. The developer never edits the CMS source code; they only edit config files.

1. **Section Definitions** (`client/src/config/sections.js`)
   - Define what sections are available.
   - Define the fields and types for each section.
   - The CMS automatically renders the correct forms.

2. **Collection Definitions** (`client/src/config/collections.js`)
   - Define what collections exist (testimonials, faqs, services, team, blog-posts).
   - Define the fields for each collection.
   - The CMS automatically renders the collection manager pages (create, edit, list, delete).

3. **Sidebar Navigation** (`client/src/config/navigation.js`)
   - Define which links appear in the sidebar and in what order.
   - This allows developers to hide features they don't need for a specific client (e.g., hide Blog for a plumber).

### 9.2 Starter Kit Workflow

1. **Clone** the `cms-agency` repo.
2. **Install** dependencies (`npm install` at root).
3. **Run** `npm run dev` (starts both client and server).
4. **Customize** `client/src/config/sections.js` and `client/src/config/collections.js` for the client's needs.
5. **Build the client's website** as a separate frontend that reads from `content/pages/*.json`.
6. **Deploy** by running `npm run build` (builds the admin dashboard SPA) and serving it with the Express server.

### 9.3 Extensibility

- **Adding a new field type:** The developer updates `client/src/components/fields/` with a new component and registers it in `fieldTypeMap.js`. The CMS UI automatically picks it up.
- **Adding a new section:** Add an entry to `sectionRegistry` and build the corresponding React component in the client's website.
- **Adding a new collection:** Add an entry to `collectionRegistry` and a new JSON file in `content/collections/`.

## 10. Security Considerations

- **No SQL Injection:** There is no database. All storage is JSON files.
- **Path Traversal:** The API must sanitize all `slug` and `filename` parameters. A strict regex (`^[a-z0-9-]+$`) is applied to all path parameters. `filename` is sanitized with `path.basename` and checked against the uploads directory.
- **Authentication:** `bcrypt` for password hashing. `express-session` with `httpOnly` cookies. CORS is restricted to the frontend origin in production.
- **File Uploads:** `multer` with `fileFilter` to accept only images (`image/jpeg`, `image/png`, `image/webp`, `image/svg+xml`). Max file size: 5MB.
- **XSS:** All JSON content is treated as plain text on the frontend (no `dangerouslySetInnerHTML` in the CMS dashboard). The client's website is responsible for sanitizing any rich text fields they render.
- **Rate Limiting:** `express-rate-limit` on the login endpoint (5 attempts per 15 minutes).

## 11. Build & Deployment

### 11.1 Development

```bash
npm run dev
# Runs: concurrently "npm run dev:server" "npm run dev:client"
```

### 11.2 Production

```bash
npm run build
# 1. Builds the React client into server/public/
# 2. The Express server serves the built SPA from server/public/
# 3. The Express server also serves the API from /api
# 4. The Express server serves content/uploads/ as static files
```

### 11.3 Deployment Targets

- **VPS / Dedicated Server:** Run `npm run build` and `npm start`. Serve behind Nginx or Caddy as a reverse proxy for HTTPS.
- **Node.js Hosting (Render, Railway, etc.):** Works out of the box. The `content/` directory must be a persistent disk or volume.
- **Static Hosting (Netlify / Vercel) for the website:** The CMS itself is a dynamic Node.js app. The client's website can be a static site built from the JSON data. The CMS and the website can be separate deployments, or the CMS can serve the website as static files from the same Express instance.
